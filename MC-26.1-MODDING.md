# Minecraft 26.1 Fabric Modding — Field Notes

Distilled from porting two mods (toms-mobs and PolyLootr) and patching one (Lootr 1.21.11). Paste into a new Claude Code session as context for any 26.1 Fabric work.

## TL;DR

- 26.1 is the first Minecraft that ships **deobfuscated** by default. Mojang dropped `client_mappings` from the version manifest. Yarn has no 26.1 build.
- Class names in the jar are neither Yarn nor Mojmap — a third naming scheme close to 1.21.11 Mojmap but drifted. Inspect with `javap` before writing mixins.
- Toolchain: **Gradle 9.2.1+, Loom 1.15-SNAPSHOT, Java 25, Mixin compat `JAVA_25`**.
- The Loom plugin id is **`net.fabricmc.fabric-loom`** (full id, not legacy `fabric-loom`).
- **Do not** add `mappings loom.officialMojangMappings()` — it throws because no Mojang mappings exist for 26.1.
- Use `implementation` (not `modImplementation`) for deps; without remapping, modImplementation does nothing useful and can confuse newer Loom.

## Reference projects

Working 26.1 Fabric mods to copy build configs from:

- [toms-mobs](https://github.com/nickisashkir/toms-mobs) — content mod with custom mobs
- [PolyLootr](https://github.com/nickisashkir/PolyLootr) — server-side Polymer wrapper, mixins, virtual entities, datapack-guard mixins
- [Fabric API 26.1 branch](https://github.com/FabricMC/fabric/tree/26.1) — ground truth for what works on 26.1

## build.gradle template

```gradle
plugins {
    id 'net.fabricmc.fabric-loom' version '1.15-SNAPSHOT'  // FULL id, not 'fabric-loom'
    id 'maven-publish'
}

version = "${project.mod_version}"
group = project.maven_group

base {
    archivesName = project.archives_base_name
}

repositories {
    maven { url 'https://maven.nucleoid.xyz' }     // Polymer
    exclusiveContent {
        forRepository {
            maven { name = "Modrinth"; url = "https://api.modrinth.com/maven" }
        }
        filter { includeGroup "maven.modrinth" }
    }
}

dependencies {
    minecraft "com.mojang:minecraft:${project.minecraft_version}"
    // NO mappings line — officialMojangMappings() throws on 26.1, no Yarn exists
    implementation "net.fabricmc:fabric-loader:${project.loader_version}"  // NOT modImplementation
    implementation "net.fabricmc.fabric-api:fabric-api:${project.fabric_api_version}"

    // Polymer (optional but common)
    implementation include("eu.pb4:polymer-core:${project.polymer_version}")
    implementation include("eu.pb4:polymer-virtual-entity:${project.polymer_version}")
}

processResources {
    inputs.property "version", project.version
    inputs.property "minecraft_version", project.minecraft_version
    inputs.property "loader_version", project.loader_version
    filteringCharset "UTF-8"

    filesMatching("fabric.mod.json") {
        expand "version": project.version,
                "minecraft_version": project.minecraft_version,
                "loader_version": project.loader_version
    }
}

def targetJavaVersion = 25
tasks.withType(JavaCompile).configureEach {
    it.options.encoding = "UTF-8"
    it.options.release.set(targetJavaVersion)
}

java {
    def javaVersion = JavaVersion.toVersion(targetJavaVersion)
    if (JavaVersion.current() < javaVersion) {
        toolchain.languageVersion = JavaLanguageVersion.of(targetJavaVersion)
    }
    withSourcesJar()
}

jar {
    from("LICENSE") {
        rename { "${it}_${base.archivesName.get()}" }   // archivesBaseName is deprecated
    }
}
```

## Naming drift from 1.21.x to 26.1

This is what consumed the most porting time. Most of these surface as compile errors, but a few are runtime failures.

| 1.21.x Mojmap | 26.1 deobf |
|---|---|
| `ResourceLocation` | `Identifier` (`net.minecraft.resources.Identifier`) |
| `Identifier.of(ns, path)` | does not exist — use `Identifier.fromNamespaceAndPath(ns, path)` |
| `AbstractMinecart` (no `.minecart.` subpkg) | now in `net.minecraft.world.entity.vehicle.minecart.AbstractMinecart` |
| `chunk.setUnsaved(true)` | `chunk.markUnsaved()` (no arg) |
| `entity.getTags()` / `getScoreboardTags()` | `entity.entityTags()` |
| `level.isClientSide` (field) | private field — call `level.isClientSide()` (method) |
| `display.setItemStack(stack)` (public) | private — use `display.getSlot(0).set(stack)` via SlotAccess |
| `display.setTransformation(t)` (public) | private — needs `@Invoker` accessor mixin |
| `nbt.getString(key)` returns `String` | returns `Optional<String>` (since 1.21.5, persists in 26.1) |
| `Player` parameter on `IHasOpeners.hasServerOpened` | unchanged but **mixin descriptors must use `Lnet/minecraft/world/entity/player/Player;`** — using `ServerPlayer` produces silent "no targets" |
| `withPropertiesOf(state)` | unchanged but only copies properties whose names match — e.g., a `facing`-property block won't pick up a `rotation` property |

## Inspecting the deobf jar

The 26.1 deobf jar lives at:
```
~/.gradle/caches/fabric-loom/minecraftMaven/net/minecraft/minecraft-merged-deobf/26.1/minecraft-merged-deobf-26.1.jar
```

Use `javap` to verify classes / methods / fields before writing mixins:

```bash
JAR=~/.gradle/caches/fabric-loom/minecraftMaven/net/minecraft/minecraft-merged-deobf/26.1/minecraft-merged-deobf-26.1.jar
javap -p -cp "$JAR" net.minecraft.world.entity.vehicle.minecart.AbstractMinecart
javap -c -p -cp "$JAR" net.minecraft.resources.Identifier   # bytecode for static factories
```

For modded mods, find the resolved jar in `~/.gradle/caches/modules-2/files-2.1/` and javap the same way:

```bash
LOOTR=~/.gradle/caches/modules-2/files-2.1/maven.modrinth/lootr/<id>/<hash>/lootr-<id>.jar
javap -p -cp "$LOOTR" noobanidus.mods.lootr.fabric.init.ModBlocks
```

A mixin that **compiles** can still **fail at load** with `InvalidInjectionException` if the method name or INVOKE descriptor is wrong. Always verify both with `javap -c -p` before writing the injector.

## Mixin gotchas

### Lambda targets are fragile

`@WrapOperation(method = "lambda$static$2", ...)` and similar synthetic-name targets break across compilations. They depend on:
- Source code order (any reorder shifts the index)
- Compiler / deobfuscator choices (Yarn often renames lambdas to `method_NNNNN`)
- A refmap (which is easy to forget to ship)

**Better pattern**: target the public field via `@Shadow @Mutable` and wrap in `<clinit>` TAIL:

```java
@Mixin(EntitySelector.class)
public class MixinEntitySelector {
    @Shadow @Final @Mutable
    public static Predicate<Entity> CONTAINER_ENTITY_SELECTOR;

    @Inject(method = "<clinit>", at = @At("TAIL"))
    private static void wrap(CallbackInfo ci) {
        Predicate<Entity> original = CONTAINER_ENTITY_SELECTOR;
        CONTAINER_ENTITY_SELECTOR = entity -> {
            if (shouldBlock(entity)) return false;
            return original.test(entity);
        };
    }
}
```

### Mixin descriptor must match exact runtime parameter type

`hasServerOpened(Player)` in source maps to `Lnet/minecraft/world/entity/player/Player;` in the descriptor. If the actual call site passes a `ServerPlayer`, your mixin **target** still uses `Player` (the declared method parameter). Getting this wrong silently produces "scanned 0 target(s)" with no clear error.

### Mixin recursion with RandomizableContainer

In any mixin that runs inside Lootr-style mods that hook `unpackLootTable`:

```java
// DON'T:
container.getItem(0)   // → calls unpackLootTable() → triggers Lootr's mixin
                       //   → calls back into your mixin → infinite recursion

// DO:
container.getLootTable()  // simple field getter, no side effects
```

This bit us hard during the waystones datapack work.

### tickElementHolder defaults to false

If you implement `BlockWithElementHolder` and override `createElementHolder` but expect `onTick()` to fire, you also need:

```java
@Override
public boolean tickElementHolder(ServerLevel world, BlockPos pos, BlockState state) {
    return true;
}
```

The default returns `false`. Easy to miss — the symptom is "particles never emit despite the holder existing."

## MixinExtras

Bundled with Fabric Loader 0.14+. `@ModifyExpressionValue`, `@WrapOperation`, `@Local(argsOnly = true)`, `@ModifyReturnValue` all work without extra deps. Don't add `mixinextras-fabric` to your build — it's already there.

## Polymer 0.16 API (verified on `0.16.0-pre.3+26.1`)

```java
// Vanilla overlay for a modded block — second arg is functional interface
PolymerBlockUtils.registerOverlay(myBlock,
    (state, ctx) -> Blocks.CHEST.withPropertiesOf(state));

// Modded BE → vanilla BE type — second arg is functional
PolymerBlockUtils.registerBlockEntity(myBEType,
    (be, ctx) -> BlockEntityType.CHEST);

// Modded item → vanilla item
PolymerItemUtils.registerOverlay(myItem,
    (stack, ctx) -> Items.CHEST);

// Modded entity → vanilla entity (note the curried function shape)
PolymerEntityUtils.registerOverlay(myEntityType,
    entity -> ctx -> EntityType.CHEST_MINECART);

// Virtual-entity overlay attached to a block (lodestone markers, item displays, etc.)
BlockWithElementHolder.registerOverlay(myBlock, myRendererSingleton);
```

### Polymer ElementHolder

```java
public class MyHolder extends ElementHolder {
    @Override
    protected void onTick() {
        var attachment = getAttachment();           // HolderAttachment
        ServerLevel world = attachment.getWorld();
        Vec3 pos = attachment.getPos();

        // getWatchingPlayers() → Collection<ServerGamePacketListenerImpl>
        for (var handler : getWatchingPlayers()) {
            ServerPlayer player = handler.getPlayer();
            // ...
        }
    }
}
```

Display element types: `BlockDisplayElement`, `ItemDisplayElement`, `TextDisplayElement`. All extend `DisplayElement` and support `setScale`, `setOffset`, `setBillboardMode`, `setTranslation`, `setRotation`.

### Per-player particle emission

Use the `ServerLevel.sendParticles` overload that takes a specific `ServerPlayer`:

```java
world.sendParticles(player, ParticleTypes.ENCHANT, false, false,
    x, y, z, count, dx, dy, dz, speed);
```

This sends to **one** player only. No need to copy/modify the broadcast version. When the Lootr dev told us "you'll need to copy `ServerLevel::sendParticles` and provide a player list" he was wrong — this overload already handles it.

### Polymer registry sync

```java
import eu.pb4.polymer.rsm.api.RegistrySyncUtils;

RegistrySyncUtils.setServerEntry(BuiltInRegistries.PARTICLE_TYPE, myParticle);
```

Marks an entry as server-only so vanilla clients don't get "unknown registry entry" warnings on connect.

**Gotcha**: when the registry's value type IS `Identifier` (like `CUSTOM_STAT`), the two `setServerEntry` overloads (`(Registry<T>, T)` and `(Registry<T>, Identifier)`) are ambiguous. Workaround:

```java
@SuppressWarnings({"rawtypes", "unchecked"})
private static void markServerOnly(Registry registry, Object entry) {
    RegistrySyncUtils.setServerEntry(registry, entry);
}

markServerOnly(BuiltInRegistries.CUSTOM_STAT, myStatId);
```

## Fabric mod load order

Mods listed in `depends` load before the dependent mod's `onInitialize` fires. Useful when you need a third-party mod's static fields to be populated before you read them — declare it in `depends` and your `onInitialize` is guaranteed to run after theirs.

If you can't rely on this (e.g., the dep doesn't populate fields in `onInitialize` but in some later lifecycle event), use a TAIL `@Inject` into the dep's main initializer instead. One mixin → guaranteed timing.

## Datapack compatibility (lessons from waystones)

Datapacks frequently use vanilla `barrel` or `chest` blocks with custom `Items` NBT as GUI containers. Mods that hook **every** vanilla container (Lootr is the canonical example) will damage these blocks unless guarded.

**Reliable check** to identify "this is a datapack custom block, not a real loot container":

```java
blockEntity instanceof RandomizableContainer container && container.getLootTable() == null
```

If a container has no loot table, it's not world-gen loot — leave it alone.

**Don't** check item contents for datapack markers (e.g., `custom_data` component). That requires calling `getItem(int)`, which calls `unpackLootTable()`, which triggers Lootr's own mixin, which infinite-recurses through your mixin. Stick to `getLootTable()`.

## Avoiding tornato-style mistakes

When porting/wrapping a mod for vanilla clients via Polymer:

1. **Always `./gradlew build` early.** Surface name drift via the compiler instead of guessing.
2. **Don't use `@ModifyArg` to intercept registration loops.** The lambda runs on EVERY registered entry — easy to hit unexpected types and ClassCastException at startup.
3. **Per-item, per-BE, per-entity mappings.** "Everything is a barrel" / "everything is a barrier" is wrong; map each modded thing to its closest vanilla equivalent individually.
4. **Verify property compatibility.** When mapping a `HorizontalDirectional` block to a vanilla equivalent, pick one that also has `facing` (e.g., `BLAST_FURNACE`), not one with `rotation` (e.g., `PLAYER_HEAD`).
5. **Don't blindly disable upstream features.** Custom stats/particles can be marked server-only via `RegistrySyncUtils.setServerEntry` instead of `@Overwrite`-ing them away.

## Filament for custom textures

For mods that need custom block textures or item models on vanilla clients:

[Filament](https://github.com/Patbox/filament) is a Polymer addon that lets the server ship custom models / textures via a server-side resource pack. No client mod needed.

Useful when Polymer's "map to nearest vanilla block" approach is too lossy (e.g., distinctive Lootr chest texture).

## Quick command reference for Lootr-using servers

When running Lootr in production:

```
/lootr custom-chest <x> <y> <z> <loot_table>   # convert a single vanilla chest
/lootr custom-map <level> <loot_table>          # bulk-convert in a dimension
/lootr clear                                    # reset opener list at targeted block
/lootr refresh                                  # regenerate loot at targeted block
```

Common loot tables: `minecraft:chests/simple_dungeon`, `minecraft:chests/abandoned_mineshaft`, `minecraft:chests/bastion_treasure`, `minecraft:chests/buried_treasure`, `minecraft:chests/end_city_treasure`, `minecraft:chests/trial_chambers/reward`. When unsure, `simple_dungeon` is the safe generic.
